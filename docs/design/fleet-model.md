# Fleet model: box identity, storage, and lifecycle

## Motivation

Harness is deployed as a fleet: many short-lived compute instances (a
sandbox, a VM, a container — this doc calls each one a **box**), each
running one `harness serve` process, coordinated by one external
orchestrator (a **hub**) that a box never needs to know the implementation
of. `server/openapi.yaml` already assumes exactly this shape ("a single
external orchestrator drives many harness instances through this API"; "the
hub holds exactly one [event stream] per box") without ever writing down
the model that makes those sentences true. This is that write-up: what
identifies a box, what survives its death, what a client can rely on across
a restart, and the one environment-variable contract a hub and a box's
deployment tooling share. It is a build spec in the same register as
`docs/design/context-compaction.md` — states, wire fields, invariants, a
test list — except most of what it specifies is **already implemented**
(session lineage and goal pausing shipped alongside this document); only
§8's hub-side half is deliberately deferred.

Written generically: no deployment-provider specifics beyond neutral
examples (a volume, a container platform, a sandbox). See
`docs/deploy-modal.md` for one concrete instantiation of this model.

## 1. Identity: the operator-chosen box NAME

A box's identity is a single string, the **NAME** — chosen by the operator
(or the hub, on the operator's behalf) at spawn time, never generated by
harness itself. Harness has no notion of "box" or "name" as first-class
concepts in its own code; the name exists entirely at the deployment layer,
as the thing that determines *which durable storage this process's session
directory points at* (see §2). Two different processes serving the same
name, at different times, are the same box's history; two processes
serving different names never share anything.

Names are operator-legible on purpose (`task-142`, `alice-refactor`, a
ticket ID) — the whole point of naming a box, rather than letting an
infrastructure layer assign an opaque instance ID, is that a human or a
fleet UI can look at a name and know what it is for.

## 2. Storage: one volume per name, single-writer by construction

Each name maps to exactly one durable volume or directory, and
`HARNESS_SESSION_DIR` (see `cmd/harness/main.go`'s `sessionDir`) is how the
box process is told where that volume is mounted. The mapping from name to
storage location is deployment-tooling's job, not harness's — harness only
ever sees the resolved directory.

**Invariant: a name's volume is never mounted read-write by two
concurrently-live `harness serve` processes.** The session log
(`engine/store.go`) is append-only and assumes a single writer per session
file; two processes appending to the same `<id>.jsonl` or `events.jsonl`
concurrently is a corruption hazard the engine does not defend against
(and should not have to — it is a deployment-topology invariant, not a
locking problem). This is why ADOPT (§4) is a full *stop-the-old, start-the-
new* handoff, never two processes sharing a volume live.

Nothing about this requires the volume to be literally one device — a
directory on shared network storage, one bucket-backed mount, or a local
disk on a persistent VM all satisfy it, as long as the single-writer
invariant holds.

## 3. Cattle vs pets

- **Boxes are cattle.** The compute — the container, sandbox, or VM
  process — is fungible and ephemeral. It has no identity of its own worth
  preserving; when it dies, nothing about *it* is missed, because everything
  that matters was never in it (see §2). A crashed or upgraded box is
  simply replaced.
- **Names and volumes are pets.** The name and the storage behind it are
  the durable unit an operator tracks, retries against, and eventually
  decides to retire. Losing a name's volume loses real history — the
  session log, the goal state, the task's provenance thread (§6) — in a way
  losing a box never does.

This is the standard cattle/pets distinction, applied specifically to the
box/name split rather than to compute-vs-data in the abstract: it is what
makes ADOPT (§4) meaningful rather than merely convenient.

## 4. Lifecycle

```
        spawn                         death
  (none) -----> ALIVE(name) -----------------------> DEAD(name), volume survives
                    |                                       |
                    | box process exits/crashes/is killed    | ADOPT: respawn same name
                    v                                        v
                 (see below)                            ALIVE(name), history restored
                                                               |
                                                               | retire
                                                               v
                                                          GONE(name), volume deleted (GC)
```

- **Spawn.** A fresh name — one that has never been used, or one whose
  volume was already GC'd (§ retire) — gets an empty session directory:
  `harness serve` boots with no sessions, `POST /session` creates the
  first one. Nothing distinguishes this from a from-scratch install; there
  is no "provisioning" state harness participates in.
- **Death.** The box process exits (cleanly via `Drain`/`Shutdown`, or
  abruptly — killed, OOM'd, the host reclaimed). The volume is untouched:
  whatever was durably flushed (every session log write, and the
  `events.jsonl` journal — see `engine/store.go` and `server/journal.go`)
  survives; nothing still pretends to be alive. There is no heartbeat or
  lease that would let a dead box's session report anything other than
  "on disk, unattended" the instant the process is gone — a
  **read of the volume from a fresh process is the only liveness truth**,
  which is exactly what ADOPT performs.
- **ADOPT.** Respawning the **same name** means the new box process's
  `HARNESS_SESSION_DIR` resolves to the **same volume** the dead box used.
  `harness serve` boots against it exactly the way it boots against any
  on-disk session directory:
  - Every session's history restores (`engine.ListSessions` +
    `LoadSession`), because it was always just files on the (surviving)
    volume — no special "recovery" code path, no export/import step.
  - Any goal that was armed (active, neither achieved nor cleared) when the
    old box died surfaces as `paused: true, pause_reason: "restart"` on
    that session's goal summary (see §7) — not silently, not as an
    indistinguishable "still running" state. The composite `state` reads
    `idle`, so a client or composer can act on the session immediately
    (send a new prompt, or one-click re-arm the paused goal via `POST
    /session/{id}/goal` with its existing condition) rather than waiting
    on a loop that will never resume itself.
  - ADOPT is *also* the binary-upgrade path: stop the box running version
    N, spawn a fresh box on the same name running version N+1 pointed at
    the same volume. There is no distinct "upgrade" procedure — a box
    restart and a binary upgrade are the same operation from the volume's
    point of view, which is why §2's single-writer invariant (never two
    live writers on one volume) is what makes upgrades safe: the old
    process must actually be gone before the new one starts.
- **Retire + GC.** When a name is done being useful (the task is finished,
  abandoned, or merged into a later `parent_session`-linked box — see §6),
  an operator deletes the name's volume. This is the only true data loss
  in the model, and it is deliberate: an operator action, not a timeout or
  an automatic sweep harness performs itself. Before this point, a name can
  always be ADOPTed again, no matter how long its volume has sat idle.

## 5. Sessions-per-box and parallelism doctrine

The engine and server support any number of sessions per box (`POST
/session` has no built-in cap beyond `Options.MaxResident`, which bounds
*resident* sessions in memory, not sessions overall — see
`server/server.go`). But the fleet's **doctrine** is one ACTIVE task per
box: parallelism comes from spawning more boxes (more names), not from
running multiple concurrently-important sessions on one. Two reasons this
is a doctrine rather than a mechanism:

- **Workdir exclusivity is per-box.** `share_workdir`/`workdir_isolation`
  (see `server/openapi.yaml`'s `createSession`) already prevent two
  sessions on the same box from silently corrupting a shared working
  directory, but they do nothing to prevent two *unrelated* tasks from
  contending for the same box's compute and attention. One active task per
  box keeps that contention question moot.
- **A box is one deploy unit.** Upgrading, restarting, or retiring a box
  (§4) affects every session on it at once. A box serving one task means
  those operations have a single, legible blast radius; a box serving many
  unrelated tasks means an operator restarting it for one task's sake
  collaterally disrupts every other task resident there.

A box MAY carry more than one session at a time in practice — a
follow-up question thread, a scratch session used for inspection — but the
fleet's unit of scheduling and parallelism is the box, not the session.

## 6. Lineage: `parent_session`

`POST /session` accepts an optional `parent_session` field: an opaque
provenance pointer to the session this one continues from — a re-dispatch
after a failed or abandoned goal, a follow-up fix picked up on a fresh box
(often a fresh NAME, via ADOPT's binary-upgrade path or simply a new box
for the next round of work on the same task). It is:

- **Set once, at creation**, and durable across resume/restart — persisted
  on the session log's header record exactly the way `WorkDir` is (see
  `engine/store.go`), restored by `LoadSession`, exposed as
  `parent_session` on `GET /session` list entries and `GET /session/{id}`
  (see `server/handlers.go`).
- **Never validated against this server's own sessions.** Lineage crosses
  boxes routinely (the whole point: a re-dispatch typically lands on a
  *different* box/name than the session it continues from), so the only
  validation is a sanity length cap (see §9) — never a "does this session
  exist" check, which would be both wrong (the parent may live on a box
  that's since been retired) and impossible in general (a parent may live
  on a box this server has never talked to).
- **The thread a fleet UI groups by.** Given a chain of sessions each
  naming the previous one as `parent_session`, a UI can render "this task's
  history" as a sequence of dispatches across however many boxes it took,
  independent of which name or volume each one happened to run on. This is
  the mechanism that lets "one active task per box" (§5) coexist with a
  task that legitimately needs several dispatches: each dispatch is a fresh
  box/session, linked by lineage rather than by being crammed onto one box.

## 7. Goal pause vocabulary

A goal (`engine.Session.PursueGoal`, see `docs/goal-loop.md`) can be
*active* with no loop currently driving it, for two structurally different
reasons — and an operator or composer needs to tell them apart, not just
know "active":

- **`pause_reason: "restart"`** — this box booted (fresh spawn or ADOPT,
  §4) and found a session's goal armed (active, neither achieved nor
  cleared) in its journal, with no loop ever attached in this process.
  Detected once, at boot (`server.pauseArmedGoalsAtBoot`), and journaled as
  a durable `goal.paused` event so the history reads honestly instead of
  silently omitting why the goal never progressed. The composite `state`
  reads `idle` (not `busy`, not `goal-running`) specifically so a client
  does not mistake this for a live loop — see `server/goal_paused_test.go`.
  Clears the instant an operator (or automation) re-arms it: `POST
  /session/{id}/goal` with the session's existing condition resumes it
  (rejecting a *different* condition with 409, rather than silently
  discarding the old goal) — a "one-click re-arm" in fleet-UI terms.
- **`pause_reason: "provider-backoff"`** — the retryable-backoff park
  machinery (`engine/goal.go`, from the retryable-provider-errors work,
  GitHub issue #61) is waiting out provider weather: a loop *is* running,
  it is just between attempts of a long, jittered backoff. The composite
  `state` stays `goal-running` here — nothing about the loop's actual
  behavior changes, only its observability (`goal_paused`/
  `goal_pause_reason` mirror the existing `retryable`/`waiting` fields).
  Clears on its own, with no operator action, the moment the loop's own
  retry succeeds — it "already re-arms" itself by construction (the same
  self-re-arm design already documented in `engine/goal.go`).

Both share one wire shape (`paused: bool`, `pause_reason: "restart" |
"provider-backoff"`) on the goal summary (`GET /session`, `GET
/session/{id}`, `GET /session/{id}/wait`) and, where applicable, on the
`goal.*` event vocabulary (`goal.paused`, and `goal.stalled` while
retryable && waiting) — see §9 and `server/openapi.yaml`.

## 8. The hub spawn contract: `HARNESS_HUB_BOX_NAME`

This section documents a contract; **it is not implemented in this repo**
(the hub itself is out of scope here and lands on another branch). It is
recorded now so the deployment side of the model in §1–§2 has one concrete
handle to build against.

When a hub spawns a box, it generates or selects the box's NAME (§1) and
passes it to the spawn command's environment as `HARNESS_HUB_BOX_NAME`.
Deployment scripts invoked by that spawn command read this variable to
derive per-name storage — e.g. mount or create the volume/directory the
name maps to (§2) — and typically also use it to set
`HARNESS_SESSION_DIR` to that resolved path before `harness serve` starts.
The two variables are deliberately separate: `HARNESS_HUB_BOX_NAME` is the
*identity* the hub asserts, `HARNESS_SESSION_DIR` is the *storage location*
deployment tooling derives from it — harness's own code only ever consumes
the latter (see `cmd/harness/main.go`'s `sessionDir`) and has no notion of
`HARNESS_HUB_BOX_NAME` at all. This split is what lets the same box image
run unmodified whether it was spawned by a hub, a human running a script by
hand (who sets `HARNESS_SESSION_DIR` directly), or a test harness.

See `AGENTS.md`'s "Fleet model (the deploy story)" section for the
short-form cross-reference back to this document.

## 9. Wire fields (summary)

```jsonc
// POST /session request body (additions)
{
  "parent_session": "ses_..."   // optional, 1-128 bytes, opaque
}

// Session (GET /session list entry, GET /session/{id})
{
  "parent_session": "ses_...",  // omitempty; absent = no recorded lineage
  "state": "idle",              // "idle" | "busy" | "goal-running"
  "goal": {
    "active": true,
    "paused": true,
    "pause_reason": "restart"   // "restart" | "provider-backoff"
    // ...existing goal fields (condition, turns, retryable, waiting, ...)
  }
}

// Event (goal.* vocabulary additions)
{
  "type": "goal.paused",        // new event type, boot-time only
  "session_id": "ses_...",
  "goal_condition": "...",
  "goal_paused": true,
  "goal_pause_reason": "restart"
}
// goal.stalled also carries goal_paused/goal_pause_reason
// (= "provider-backoff") while goal_retryable && goal_waiting.
```

## 10. Invariants

- A name's volume is never mounted read-write by two concurrently-live
  `harness serve` processes (§2) — ADOPT is stop-then-start, never overlap.
- A box has no persistent identity of its own; every fact worth keeping
  lives on the name's volume, never in the box's compute (§3).
- Box death never fabricates liveness: a dead box's sessions read exactly
  what the volume says, the instant a new process reads it — no lease, no
  heartbeat, no grace-period "maybe still alive" state (§4).
- An armed goal with no attached loop is never presented as
  `goal-running`/`busy` forever — restart-pause forces the composite state
  to `idle`; only a genuinely running loop (including a provider-backoff
  park, which IS a running loop) reads `goal-running` (§7).
- `parent_session` is never dereferenced or validated against this
  server's own session set — only a length/non-emptiness check (§6).
- `HARNESS_HUB_BOX_NAME` is a deployment-environment contract only; harness
  code never reads it (§8).
